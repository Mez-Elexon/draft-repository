1. Architecture

2. Object Model

The dispatch api uses the following objects:

  • program

  • event

  • report

  • ven

  • subscription

2.1. Program

A program represents a relationship between an operator and a supplier. It is the main object that all events (or dispatches) are related to. The program is created and maintained by the operator and can be read by the supplier.

2.1.1. Lifetime

The period of time that a program is active for is determined by its intervalPeriod. This is represented by a start and duration. The start is the time that the program is created and the duration is the length of time that the program is active for.

2.1.2. Targetting

The program will be targetted at a VEN by the use of a target type of VEN_NAME. The value of the target will be the venName of the VEN.

2.1.3. Multiplicity

There can be multiple programs with the same target VEN_NAME`. This allows for the operator to have multiple programs with the same VEN.

2.2. Event

An event is a dispatch that is linked to a program. A dispatch can be a realtime or a scheduled dispatch. All events will have a starttime and a duration. A starttime of 0000-00-00 will indicate that the event is a realtime dispatch. Only an operator can create and manage an event. A supplier can only read events relating to programs that they are targetted at.

2.3. Report

A report is an object that a supplier creates in response to an event. A report must be linked to an event and therefore a program. The report will be read by the operator.

2.4. ven

A ven is an object that represents the VEN client. It will be created initially by the operator and then maintained by the supplier. The VEN will be required to update the ven object with its current status regularly.

2.6. Object Identifiers

All objects will have a unique identifier. The identifier will be generated by the VTN when the object is created. The object identifier will have a 4 character prefix to namspace the object to a specific VTN. The object identifier will be a UUID. The 4 character prefix known as the vtn namepsace will be assigned to the VTN and be unique to the VTN. The remaining characters will be a UUID aligned with RFC 9562.

5. Dispatch Details

A dispatch event can be either a realtime dispatch or scheduled. A scheduled dispatch can be either a self-dispatch or a confirm-dispatch variant.

5.1. Self Dispatching

An event that is considered self-dispatch will have a payload item SELF_DISPATCHED set to true. This indicates that the VEN will dispatch when the interval period starts. If this payload item is set to false, then it indicates that the dispatch is for information only and should not be dispatched until confirmation is received. Confirmation is received by updating the event to have a SELF_DISPATCHED payload of true. This transforms the event into a self-dispatch event and will execute according to these rules. Note that if no confirmation is received before the interval period start, then the event is considered lapsed.

Self Dispatched
{
  "payloadDescriptors": [ {
    "payloadType": "SELF_DISPATCHED"
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "SELF_DISPATCHED",
      "values" : [ true ]
    } ]
  } ]
}
The SELF_DISPATCHED payload item is a boolean value and must be declared in the payloadDescriptors and intervals.

5.2. Dispatch Period

A dispatch consists of one or more intervals. An interval can define a interval period that consists of a start time and duration. If an interval dioes not include a period then the event level interval period is applied. In a very simple case a dispatch with a single interval with start at the event level start time and last for the duration defined at the event level. See section on multiple intervals for more information.

Realtime Dispatch
{
  "intervalPeriod" : {
    "start" : "0000-00-00",
    "duration" : "PT3H"
  }
}
Scheduled Dispatch
{
  "intervalPeriod" : {
    "start" : "2025-04-12T04:00:00Z",
    "duration" : "PT3H"
  }
}
Special values

Start values of 0000-00-00 and 0000-00-00T00:00:00 are considered do it now. A duration of P9999Y is considered forever.

5.3. Dispatch Targets

The event is linked to a program which is used to target the dispatch to a VEN. The event can also define targeting information that the VEN can use to understand which assets should be dispatched. There will be 2 types of targets: ASSET_ID and ASSET_GRP. As with all targeting information, the values can be a single item or multiple. An operator can therefore dispatch to multiple assets either by specifying the asset ids or by specifying the asset group(s).

single asset id dispatch
{
  "targets" : [{
    "type" : "ASSET_ID",
    "values" : ["ASS001"]
  }]
}
multiple asset id dispatch
{
  "targets" : [{
    "type" : "ASSET_ID",
    "values" : ["ASS001", "ASS002"]
  }]
}
single asset group dispatch
{
  "targets" : [{
    "type" : "ASSET_GRP",
    "values" : ["AGRP001"]
  }]
}
multiple asset group dispatch
{
  "targets" : [{
    "type" : "ASSET_GRP",
    "values" : ["AGRP001", "AGRP002"]
  }]
}
Management of asset groups and ids

The management of asset groups and ids is the responsibility of the operator. The VTN will not validate this information. It is the responsibility of the operator to ensure that the asset groups and ids are correct.

5.4. Dispatch Payload

The information relating to the dispatch can take multiple forms.

5.4.1. Set Point Direction

A dispatch can be a request for an absolute amount DISPATCH_SETPOINT or for a relative adjustment DISPATCH_SETPOINT_RELATIVE. A positive value for the absolute amount will indicate generation and a negative value will indicate consumption. Where a relative adjustment is used, a positive value will increase the current setpoint and a negative value will decrease.

If the current setpoint is -100 and the adjustment is positive 10, then the resource is still consuming power but at a lower level.

5.4.2. Units

All dispatch values will be specified in MW. Smaller and larger magnitudes will be represented as multiples and fractions. e.g. 1GW = 1000MW and 1KW = 0.001MW. The unit will be specified in the payloadDescriptors.

5.4.3. Examples

Generate 100MW
{
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT",
    "units": "MW"
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 100 ]
    } ]
  } ]
}
Consume 10.5MW
{
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT",
    "units": "MW"
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ -10.5 ]
    } ]
  } ]
}
Adjust current setpoint by -0.5MW
{
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT_RELATIVE",
    "units": "MW"
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT_RELATIVE",
      "values" : [ -0.5 ]
    } ]
  } ]
}
An adjustment can change the sign of the setpoint. For example, if the current setpoint is +10 and the adjustment is -20 then the resource will change from generation to consumption.

5.4.4. Multiple Intervals

The examples above all refer to a single interval which by default represents the entire duration of the dispatch specified by the event’s interval period. It is possible to specify multiple intervals within the dispatch. This can be used to specify different setpoints for different periods of the dispatch. There is no limit to the number of intervals that can be specified. Intervals can be a combination of absolute and relative setpoints.

A more complex dispatch
{

  "intervalPeriod" : {
    "start" : "0000-00-00",
    "duration" : "PT1H"             (1)
  },
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT_RELATIVE",
    "units": "MW"
  },{
    "payloadType": "DISPATCH_SETPOINT",
    "units": "MW"
  }],
  "intervals" : [ {
    "id" : 0,                       (2)
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 10 ]
    } ]
  },{
    "id" : 1,                       (3)
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT_RELATIVE",
      "values" : [ -5 ]
    } ]
  },{
    "id" : 2,                       (4)
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 10 ]
    } ]
  } ]
}
1 The interval period is 1 hour and therefore the event will last 3 hours
2 The first interval sets the setpoint to 10MW
3 The second interval adjusts the setpoint by -5MW
4 The third interval sets the setpoint back to 10MW
Event length

The length of an event is the sum of all the intervals. An event with a single interval is simply the interval period. Where there are multiple intervals then each interval will use the duration of the interval period.
Variable Interval Periods

It is possible to have variable interval periods. This can be achieved by including the optional intervalPeriod attribute with the interval itself. This will override the intervalPeriod of the event. This can be used to create a dispatch that has different interval periods within the event.

A more complex dispatch
{
  "intervalPeriod" : {
    "start" : "0000-00-00",
    "duration" : "PT1H"             (1)
  },
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT",
    "units": "KW"
  }],
  "intervals" : [ {
    "id" : 0,                       (2)
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 10 ]
    } ]
  },{
    "id" : 1,                       (3)
    "intervalPeriod": {
      "start": "0000-00-00",
      "duration": "PT2H"
    },
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 20 ]
    } ]
  } ]
}
1 The event level interval period is 1 hour
2 The first interval sets the setpoint to 10KW
3 The second interval sets the setpoint to 20KW and will start 1 hour after the dispatch starts. This will continue for 2 hours making the entire dispatch last 3 hours.

5.5. Dispatch Confirmation

Each dispatch event will be confirmed by the VEN with a read-back report. The report will read back the payload items that the VEN understood. A VEN will be required to send this report when they have received the dispatch event (regardless of when the event will be dispatched) and after any update.

A dispatch event including a reportDescriptor for the read-back report

    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 10 ]

}

5.6. Dispatch status report

A dispatch event will define a requirement for the VEN to create a report that confirms that the dispatch has started. The dispatch event defines a reportDescriptor with a payloadType of DISPATCH_STATUS indicating that the report will contain the status of the dispatch. The combination of the interval and the value of the historical flag lets the VN know that the report is required at the start of the first interval.

A dispatch event including a reportDescriptor for the dispatch status report
{
  "reportDescriptors": [{
    "payLoadType": "DISPATCH_STATUS", (1)
    "startInterval": 0,               (2)
    "historical": false               (3)
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ 10 ]
    } ]
  } ]
}
1 The DISPATCH_STATUS payload type is used to indicate that the report will contain the status of the dispatch.
2 A start interval of 0 indicates that the report is required for the first interval.
3 The historical flag is set to false to indicate that the report is required at the start of the interval.

The corresponding report will contain the status of the dispatch.

6. Availability

A VEN will be required to update the ven object (in the VTN) on an agreed frequency. The modification datetime of the ven object will be used to understand the liveliness of the VEN. The VEN can choose to declare itself unavailable by updating the status accordingly.

The VEN will also maintain the interval attribute to indicate how often it will update the VTN. This will be used by the business logic to determine any issues with the VEN’s availability.

VEN object
{
  "id": "26ecf048-8990-4945-803c-78eed1e644f0",

  "objectType": "VEN",
  "venName": "VEN001",
  "attributes": [ {
    "type" : "VEN_STATUS",
    "values" : [ "AVAILABLE" ]
    }, {
    "type" : "UPDATE_INTERVAL_SECONDS",
    "values" : [ 5 ]
    } ]

7.1. Enrollment

  1. For a DNO to communicate with a VEN, the VEN must be enrolled in the system. This will involve the creation of a ven object in the VTN. The ven object will declare the name of the ven and any targeting attributes.

  2. The DNO will then issue a program targeted at the VEN by using the declared targeting attributes.

  3. The VEN will use the targeting information to poll for programs. VEN will subsequently poll for events from this program.

7.2. Realtime Dispatch

  1. A DNO will create an event linked to the program used in the enrolment. The event will have an interval period with a start of 0000-00-00 to specify that the event is a realtime dispatch. It will also include an eventName of REAL_TIME_DISPATCH that will be verified against the expected parameters (e.g., a real-time dispatch will have a start of 0000-00-00).

7.5. Scheduled Dispatch - Confirmation Dispatched

This is the same as the self-dispatched event with the SELF_DISPATCHED payload item set to false. This indicates that the VEN should not dispatch until confirmation is received. Confirmation is received by updating the event to have a SELF_DISPATCHED payload of true. This transforms the event into a self-dispatch event and will execute according to these rules. Note that if no confirmation is received before the interval period start, then the event is considered lapsed.

{
  "id" : "2744e888-e270-4144-9024-cd5a279e26ec",
  "createdDateTime" : "2025-03-12T09:20:01Z",
  "modificationDateTime" : "2025-03-12T09:20:01Z",
  "objectType" : "EVENT",
  "eventName" : "SCHEDULED_DISPATCH",
  "programID" : "a816f62a-a474-4fd2-8197-f82f47d91152",
  "targets" : null,
  "intervalPeriod" : {
    "start" : "2025-04-12T04:00:00Z",
    "duration" : "PT3H"
  },
  "payloadDescriptors": [{
    "payloadType": "DISPATCH_SETPOINT",
    "units": "MW"
  }, {
    "payloadType": "SELF_DISPATCHED"
  }],
  "intervals" : [ {
    "id" : 0,
    "payloads" : [ {
      "type" : "DISPATCH_SETPOINT",
      "values" : [ "1.2" ]
    },{
      "type" : "SELF_DISPATCHED",
      "values" : [ false ]
    } ]
  } ]
}

  1. BL posts a schedule event that is SELF_DISPATCHED: false.

  2. The VEN receives the event.

  3. BL updates the existing event to SELF_DISPATCHED: true.

  4. VEN receives the update and executes on the start of the interval period.

7.6. Update Dispatch

Update a dispatch event varying aspects of the payload or definition. This is achieved using a PUT request for an existing dispatch.

  1. BL posts an event.

  2. BL issues a PUT for the existing event.

7.7. Cancel Dispatch

An existing dispatch can be cancelled while it is executing (after the start of the event) or before this time if it is a scheduled dispatch.

  1. BL schedules a dispatch.

  2. Dispatch received by VEN.

  3. BL issues a DELETE of the existing dispatch.

  4. The VEN polling for events will compare these with the existing set and determine that the event has been deleted.

7.8. Dispatch Start Event

For a realtime dispatch and scheduled self-dispatch, when the dispatch has actually started, the VEN will issue a report with this detail. This is a simple report that simply confirms execution of the dispatch.

  1. BL posts an event for a scheduled dispatch.

  2. VEN receives the event.

  3. When the start time is met and VEN executes the dispatch, it will post a dispatch started report.

  4. BL polls for reports.

Example report. The reportID refers back to the event that represents the dispatch instruction. The report includes a single resourceName and a payload that refers to the first interval.

8. Security

Security is a key concern for the VTN. The VTN is a multi-tenant system and as such it is important to ensure that data is kept secure and that tenants are not able to access data that they should not have access to. The VTN will use OAuth2 for authentication and authorization. The VTN will also use scopes to define the level of access that a client has.

All endpoints will require a valid access token to be passed in the Authorization header. The access token will be validated by the VTN and the scopes will be checked to ensure that the client has the required access.

All traffic to the VTN will be encrypted using TLS. The VTN should only be accessible via HTTPS.

8.1. High-level scoping

General, high-level security scopes are defined and follow the openADR guidelines. The following scopes are defined:

  • write_program

  • write_event

  • write_report

  • write_ven

  • read_all

In addition to these an update_ven scope is defined to allow the VEN to update its own status.

8.2. Object ownership

Programs are targetted at VENs by the use of the VEN_NAME target. A VEN will also need to read programs events. A VEN will only be able to read a program and associated events if the VEN is targetted by the program. It therefore follows that a VEN will only be able to create reports for the events that they can read.

8.3. Obtaining a token

In order to use the api, any client must request a bearer token from the OSuth2 provider. The reference implementation uses keycloak as the OAuth2 provider. The client must request a token from the OAuth2 provider and pass this token in the Authorization header of the request. The token will be validated by the VTN and the scopes will be checked to ensure that the client has the required access.

Example 1. Request Example
POST https://example.com/api/v1/resource HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

grant_type=value1&client_id=value2&client_secret=value3
Request Description

  • grant_type: The grant type requested. client_credentials.

  • client_id: The client identifier.

  • client_secret: The secret associated with the client_id

The response will be a JSON object containing the access token and the token type. Subsequent requests to the VTN will require the access token to be passed in the Authorization header.
